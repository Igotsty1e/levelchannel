# SaaS оферта — owner decisions required before publication

Legal-rf-router → legal-rf-commercial draft saved at `docs/legal/saas-drafts/saas-offer-draft-v0-pre-qa.md` (708 lines). Partner-level review (`legal-rf-qa`) pending. Before that runs, owner needs to decide:

## 1. Architectural shift — 152-ФЗ-роль

**Default plan was:** "we are the only operator of учеников ПД, teacher не оператор" — used as landing selling-point.

**Legal-rf вывод [M]:** конструкция "единственного оператора" не выдерживает 152-ФЗ при наличии teacher-edit-learner UI (`lib/auth/teacher-learner-mutations.ts`). Рекомендуется модель **"оператор + поручение"** (152-ФЗ ст. 6 п. 3):
- Платформа — единственный оператор.
- Учитель — обработчик ПДн ученика **по поручению Платформы** в пределах личного кабинета.
- Уведомление в РКН — только от нас.

**Что это меняет:**
- ✅ Marketing-claim "152-ФЗ снимаем с учителя" остаётся валидным (учитель не оператор, обязанности по 152-ФЗ — у Платформы).
- ➕ NEW документ: **Приложение №1 «Условия поручения оператора учителю»** на `/saas/processor-terms` (отдельный `doc_kind='saas_processor_terms'`).
- ➕ NEW обязательство: **audit trail каждого teacher-write события** на learner ПДн в БД (отдельная таблица + миграция; сейчас только console.info).

**Decision required:**
- [ ] Подтвердить модель "оператор + поручение" (вместо чистого "единственного оператора"). Если нет — нужен план "совместных операторов".
- [ ] Готовы добавить `/saas/processor-terms` + `saas_processor_terms` doc_kind в Sub-A.2-3-5 bundle (+1 миграция edit + +1 admin tab + +1 publish flow)?
- [ ] Готовы добавить teacher-write audit table в отдельной миграции (вне scope текущей волны, но обязательно ДО публичного запуска оферты)?

## 2. Возвраты — Apple-style + ЗоЗПП cooling period

**Owner intent был:** без возврата (Apple-style).

**Legal-rf вывод [H]:** полный no-refund **ничтожен** для B2C по ст. 16 ЗоЗПП + ст. 32 ЗоЗПП императивно даёт право отказа. Что работает:
- За уже истёкший / текущий оплаченный период — **не возвращается** (услуга считается оказанной).
- Будущие периоды — **отказ от автопродления** = нет следующего списания.
- **14-дневный cooling period** для B2C-потребителя (физлицо без статуса): если списание произошло и пользователь не пользовался платформой — возврат за невыюзанный период за вычетом фактических расходов.

**Decision required:**
- [ ] Подтвердить 14-дневный cooling-period для физлиц без статуса. Альтернатива: ограничить регистрацию ТОЛЬКО для ИП/самозанятых (тогда ЗоЗПП не применяется и no-refund работает).
- [ ] **Если выбираем cooling period** — нужна техническая реализация (на каком этапе Epic 4-DEFERRED? отдельный pull-request "refund flow"?).

## 3. Operator-managed — агентский по ГК 1005

**Owner intent был:** "мы держим деньги учеников, удерживаем %, выплачиваем учителю".

**Legal-rf вывод [M]:** это **агентский договор по ГК 1005** (модель «комиссионер»), **НЕ платёжный агент 161-ФЗ / 103-ФЗ**. Лицензия не нужна, но:
- Учителя на Operator-managed **должны быть ИП или самозанятыми (НПД)**. Платформа обязана проверять статус через API ФНС.
- Учитель-физлицо без статуса → Платформа становится налоговым агентом по НДФЛ + страховым взносам (это технически и налогово тяжело).

**Decision required:**
- [ ] Подтвердить ограничение Operator-managed только для ИП/НПД-учителей. Регистрация физлица без статуса на Operator-managed → отклонение с подсказкой "оформите НПД".
- [ ] Подтвердить интеграцию с API ФНС для проверки НПД-статуса (NEW сервис, +1 миграция).
- [ ] **Comission cap:** в оферте draft указал "не более 10%", конкретный % в Прейскуранте с правом изменения по 30-дневному уведомлению. Подтвердить, либо указать другую цифру/диапазон.

## 4. Подсудность — раздельная B2B vs B2C

**Owner intent был:** Челябинск (как в текущей оферте).

**Legal-rf вывод [H]:** для B2C-потребителя подсудность Челябинска **ничтожна** (ст. 17 ЗоЗПП + ст. 29 ГПК — выбор потребителя). Что работает:
- **B2B (учитель = ИП/самозанятый):** АС Челябинской области — это держится.
- **B2C (учитель = физлицо без статуса):** подсудность по месту жительства потребителя или месту нахождения исполнителя — по его выбору. В оферте указать честно.

**Decision required:**
- [ ] Подтвердить раздельную подсудность B2B/B2C в оферте.

## 5. Recurrent autorenew — отдельный consent

**Owner intent был:** один consent при регистрации.

**Legal-rf вывод [H]:** для recurrent-списаний нужен **отдельный явный consent** (ст. 16.1 ЗоЗПП + Указание ЦБ РФ по подпискам). Отдельный чекбокс на этапе оформления подписки.

**Что это меняет:**
- ➕ NEW `doc_kind='recurrent_subscription'` (ConsentKind тоже).
- ➕ Sub-A.2-3-5 bundle расширяется: НЕ только на этап /register, но и на этап покупки подписки в Sub-A.2 (когда self-serve recurrent flow появится в Epic 4-DEFERRED).

**Decision required:**
- [ ] Подтвердить отдельный consent на recurrent. Альтернатива — не делать recurrent на нашей стороне (CloudPayments handle всё сами + один consent у них). Это меняет архитектуру.
